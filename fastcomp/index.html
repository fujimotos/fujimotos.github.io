The key concept

The key concept of fastcomp is edit pattens. As an example, consider the statement: LevenshteinDistance(str1, str2) == 1. This statement can be rephrased as "We can transform str1 to str2 with one insertion, deletion or substitution." This means that we can verify the original statement by checking if applying any of these edit patterns (insertion, deletion, or substitution) to str1 will produce str2.

This is exactly how fastcomp measures the edit distance between input strings. Internally, fastcomp holds the computational models of possible edit patterns, and try to apply each pattern to the inputs. Why is it fast?

So how does it make the computation fast? The crucial point is that we can reject the most of edit patterns before actually applying them. Back to the diagram above. Observing that the input words 'kitten' and 'bitten' have the same string length, we can trivially conclude that one insertion (or deletion) will never transform the former into the latter. Therefore, in this case, all we need to check is that substitution one letter of 'kitten' would produce the word 'bitten'.

This argument can be easily generalized and using it we can reduce the possible edit patterns dramatically. Here we denote by w the difference in length between the input strings and by md the maximum Levenshtein distance to consider.
